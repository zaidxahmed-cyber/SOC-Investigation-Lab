# Incident Report 04: Security Log Cleared

**Detection rule:** Security Log Cleared (T1070.001)
**Severity:** High
**Status at triage:** Active, Unassigned
**Analyst:** Zaid Ahmed
**Report date:** July 2026

---

## 1. Executive Summary

A high-severity alert fired when the Windows Security event log was cleared on the workstation `DESKTOP-NV93RGT`. Clearing the Security log is a deliberate anti-forensic action used to destroy evidence, with almost no legitimate reason on a normal endpoint. This is treated as a strong indicator of malicious activity or an attempt to cover tracks, and requires immediate investigation of what happened on the host before the log was wiped.

## 3. What Triggered the Alert

The detection rule fires on Event ID 1102 ("the audit log was cleared"). This event is generated by the act of clearing the Security log itself, so the action cannot be performed silently. Because clearing the Security log destroys evidence and has no routine business purpose on a workstation, the detection is high-fidelity with a very low false-positive rate.

## 4. Investigation

### 4.1 Entities involved
- **Host:** `DESKTOP-NV93RGT` — the workstation whose Security log was cleared.
- **Account:** the account that performed the clear (captured in the event), which requires administrative rights to do.

### 4.2 Key evidence
- Event 1102 recorded on the host.
- The action requires elevated privileges, so the responsible account had administrative access.

### 4.3 Analytical reasoning
- **This is an evidence-destruction action.** Unlike most detections, there is little ambiguity about the technique: someone deliberately wiped the Security log. The question is not *what* happened but *why*, and *what it was hiding*.
- **A legitimate clear and a malicious clear look identical in the event.** The event does not say whether it was authorised. That determination comes from context: was there a scheduled maintenance activity, or does this coincide with other suspicious signals?
- **Centralised logging defeats the intent.** The attacker's goal in clearing the log is to remove evidence. Because this environment forwards logs to the SIEM in near-real-time, events written before the clear are already safe in Sentinel and remain available for investigation even though the local log is gone. This is the core defensive value of centralised logging.

### 4.4 Questions a full investigation would pursue
- Which account cleared the log, and was it authorised to do maintenance at that time?
- What activity occurred on the host in the window *before* the clear? (Investigated using the SIEM copy of the logs, not the local log.)
- Are there other suspicious signals around the same time (new accounts, privilege changes, suspicious process execution) that the clear may have been trying to hide?

## 5. Assessment

Assessed as a **true positive** and a high-priority event. Regardless of ultimate intent, a Security log clear on a workstation is abnormal and must be treated as potential evidence destruction until proven otherwise. The correct posture is to assume the host may be compromised and to reconstruct pre-clear activity from centralised logs.

## 6. Response and Recommendations

**Immediate:**
1. Identify the account that cleared the log and determine whether the action was authorised.
2. If unauthorised, treat the host as potentially compromised and begin containment.

**Follow-up:**
3. Reconstruct host activity before the clear using the SIEM's retained copy of the logs.
4. Correlate with other detections around the same timeframe to determine what the clear may have been concealing.

**Hardening:**
5. Restrict which accounts can clear event logs.
6. Confirm near-real-time log forwarding is in place so that clearing the local log does not remove evidence from the SIEM (already the case in this environment).

## 7. Note on Environment

This incident was generated in a controlled detection-engineering lab. The log was cleared intentionally using `wevtutil cl Security` to validate the detection. The investigation is written as it would be performed against a real alert.
